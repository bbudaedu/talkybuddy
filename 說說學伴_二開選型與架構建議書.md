## a. 一頁結論

### 總判定：邊緣三段（ASR/LLM/TTS）用「自建薄管線 + 抽換元件」，雲端助教層用「自建無框架 + Bedrock 原生 API」，只有「協定/介面設計」層級抄現成專案，沒有任何一層整包 fork。

理由一句話：本專案的硬約束是 push-to-talk 半雙工單機、4GB RAM、Yocto aarch64、28 天工期——所有現成框架的核心賣點（全雙工打斷、多傳輸、通用聊天記憶、個人助理生態）在這個場景下用不上，但框架重量（版本迭代風險、學習曲線、依賴衝突）是實質成本。真正該借的是「協定 schema」「插件介面設計」「工程實作細節」這類**設計資產**，不是整包程式碼。

### 二開 vs 自建，逐層總表

| 層級 | 判定 | 說明 |
|---|---|---|
| 語音管線骨架（VAD→ASR→LLM→TTS 編排） | **自建** | Pipecat/LiveKit/TEN 全部淘汰；push-to-talk 是對打斷/全雙工問題的架構性迴避，這些框架的核心價值用不上 |
| 整機語音玩具（xiaozhi-esp32-server 等） | **不 fork，抄設計** | 花 0.5-1 天讀 xiaozhi 的 WebSocket 協定 schema 與 ASR/LLM/TTS provider 插件介面，套用到自建模組邊界；記憶系統/鷹架引擎/評分/安全層仍要重寫 |
| ASR/TTS 引擎 | **元件庫替換** | 不是「要不要用框架」而是「用哪個函式庫」，sherpa-onnx 是 Apache-2.0 純函式庫，不綁架構，可無痛嵌入自建狀態機 |
| LLM 推理框架 | **自建（維持 llama.cpp）** | Ollama/mlc-llm/ExecuTorch 皆非現在該換的方向 |
| 邊緣編排/決策層 | **自建規則引擎** | LangGraph/PydanticAI 這類重量編排框架不進邊緣即時迴路 |
| 雲端助教層 | **自建無框架** | Hermes Agent/mem0/LangGraph 全部在這個資料規模下用不到；Letta 條件備選 |
| 教師儀表板 | **自建沿用現行** | 不換 Studyield，現行 vanilla JS+SVG 已符合離線可靠、零建置需求 |

## b. 對使用者外部建議的逐項評判

> 使用者原始參考來源已查明：Pipecat／Studyield 建議出自 `GitHub二開專案推薦.md`；Hermes Agent 出自 `說說學伴_28天決賽MVP規劃書.md` 與 `說說學伴_技術SPEC_v2.md`。PydanticAI／LangGraph／gstack／Superpowers 為使用者另外提出、未見於現有專案文件，依研究報告與專案脈絡逐項評判如下。

| 候選 | 裁決 | 理由 |
|---|---|---|
| **Pipecat** | **不採納**（作為骨架） | push-to-talk、半雙工、單機單使用者的場景，正是對 Pipecat 招牌能力（`InterruptionFrame` 打斷、Smart Turn 語意斷句、多 transport）的架構性迴避。官方 push-to-talk 範例本質是「關掉它最有價值的部分」，`LocalAudioTransport` 路徑還有已知半雙工閘控 bug（Issue #2941）。`GitHub二開專案推薦.md` 建議的理由是「VAD 與打斷機制內建、無需手動開發音訊對齊」——但這正是我們用不到的那部分。**28 天內自建 300 行等級狀態機的風險低於引入一個 API 仍在快速迭代（一年 113 個 release）的框架。** 唯一保留價值：其 Interruption Frame 設計思路可作未來若要做全雙工/喚醒詞升級時的參考架構。 |
| **Hermes Agent** | **不採納** | 這是本次評判中風險最高的一項，因為它**已寫入現行規劃文件**（`說說學伴_28天決賽MVP規劃書.md` §3、`說說學伴_技術SPEC_v2.md` §3.3）作為雲端助教大腦主線，必須明確裁決並修訂。經 GitHub API 與原始碼實查：Hermes Agent 確實存在、Bedrock 原生整合確實紮實（`bedrock_adapter.py`、cron scheduler 都是真的），但**架構定位是「個人在自己機器上用的自我進化助理」（Electron 桌面 app＋多聊天平台 gateway＋單機 profile），不是「後端服務同時管理數百個學生獨立診斷 agent」的多租戶型態**。要塞進本專案，得為每個學生跑一個獨立 process，維運成本隨學生數線性增加；且 53MB monorepo、25k+ open issues，28 天（其中僅 5 天要接通 demo）內二開這個陌生 codebase 的投資報酬率遠低於直接用 boto3 呼叫 Bedrock Converse API。**兩份既有規劃文件（28天規劃書 §3、SPEC v2 §3.3）需要同步修訂**，改採無框架方案（見 §d）。 |
| **PydanticAI** | **部分採納** | 邊緣層不用：1.5-1.7B 模型的原生 function-calling live accuracy 僅 58-65%、重複測試一致性可差 17 倍，用它包 tool-calling 在兒少安全攸關場景不可靠；邊緣結構化輸出改用 llama.cpp 原生 GBNF grammar（格式正確率近 100%，語意正確性交給規則引擎把關）。雲端助教層**可用**：接 Bedrock 上的 Claude，大模型原生 tool-calling 可靠度足夠支撐其驗證/重試模型，適合定義診斷報表的結構化輸出 schema。 |
| **LangGraph** | **部分採納** | 邊緣即時迴路不用：其核心賣點（durable execution、checkpoint 持久化）是為跨會話存活/可回放設計，社群已有實測案例顯示疊加即時語音會把延遲從 2-3 秒推到 5-8 秒，遠超本專案單輪 <2s 的預算，屬過度設計。雲端助教層是合理落點，但**現階段也不急著導入**——診斷邏輯目前是「一次 Bedrock tool use 呼叫就能搞定」的規模，LangGraph 的 graph/state 抽象是提前優化；建議列為 Tier 3 觀察備選，若診斷邏輯後期真的長出「先分類再依類型分支處理」的多步驟工作流，屆時用其 checkpoint 機制一魚兩吃（順便當持久記憶），現在先用無框架方案。 |
| **gstack** | **採納**（但需澄清定位） | gstack 不是產品技術棧候選，是本專案開發流程強制工具（`CLAUDE.md` 已列為必裝、`PLAN.md` 已有 GSTACK REVIEW REPORT 跑過 Eng/Design Review）。它與六份研究報告評估的語音/AI 技術選型不在同一個決策維度上，不衝突，繼續依現有規範使用（`/qa`、`/ship`、`/review`、`/investigate` 等 code review/發布關卡)。不需要、也不應該因為本次選型研究而變動。 |
| **Superpowers** | **不採納（本次不引入）** | 六份研究報告皆未涉及，性質上也是開發流程/Claude Code skill 工具而非產品架構選型，不影響邊緣或雲端技術棧的任何決策。28 天工期緊繃，團隊已有 gstack 作為品質關卡，若無具體待解決的開發流程痛點（例如缺乏 brainstorming/TDD skill 導致的實際問題），現階段不建議額外引入新工具鏈增加學習成本。若團隊評估後認為有明確效益，屬於「與本次技術選型無關」的獨立決定，可自行評估導入，不影響本建議書任何一項技術判斷。 |

## c. 修訂後系統架構

### 三層架構文字說明

1. **邊緣即時層（Genio 520，離線可用，永遠負責即時對話）**——自建 FastAPI + asyncio 狀態機驅動 push-to-talk 半雙工生命週期（IDLE→LISTENING→PROCESSING→SPEAKING→ERROR/兜底）。麥克風輸入經 silero-vad 端點偵測後，交給 sherpa-onnx + SenseVoice-Small 做 ASR（int8，CPU），OpenCC `s2twp` 簡轉繁；文字先過**規則式鷹架引擎**決定下一個教學動作，再交給 llama.cpp（Qwen3-1.7B 或保底 Qwen2.5-Taiwan-1.5B）用 GBNF grammar 約束生成一句潤飾話術（`max_tokens` 40-60、temperature 0.3-0.5）；生成結果過輕量安全過濾器，不合格即用規則引擎預存話術頂上；最終文字交給 sherpa-onnx（載入 Piper `zh_CN-huayan`/`en` voice）做 streaming TTS 分段合成播放，段間用 30-50ms crossfade 取代硬切靜音。全流程互動記錄寫入本地 SQLite。目標 TTS 首音 <300ms、單輪 <2s、端到端 <1.5s。
2. **雲端助教層（連線時，非同步 best-effort）**——本地背景程序把 SQLite 佇列中的衍生文字與分數（不含原始音訊）以 HTTPS 上傳；雲端由 EventBridge Scheduler 定期觸發 Lambda，內部用 boto3 `bedrock-runtime.converse()` 搭配強制 `toolChoice` 呼叫 Bedrock 上的 Claude，產出結構化診斷 JSON（四維評分、優弱勢、情緒狀態、教師三欄建議），寫入 DynamoDB。斷線時全部留在邊緣 SQLite，復連後補送；儀表板讀不到雲端資料時自動切換到預先準備的靜態 mock 診斷 JSON。
3. **教師端層**——沿用現行 vanilla JS + SVG 儀表板，5 秒輪詢讀取雲端/mock 診斷資料，呈現四維雷達圖、14 天趨勢、學生檔案卡。
