# 邊緣趣味互動層：小螢幕表情臉 + 氛圍燈方案評估（Genio 520 / Yocto）

研究日期：2026-07-09（多來源網路查證 + 對抗式驗證：23 來源、95 claims、驗證 25 條 → 22 確認 / 3 否決）
研究範圍：替無螢幕教學語音玩偶「說說學伴」加一塊小螢幕做**呼吸燈/氛圍、企鵝表情臉動畫、極簡觸控小遊戲**，在 MediaTek Genio 520 + **Yocto Linux aarch64** 上的完整解決方案與技術棧選型。
定位：此為**加值趣味層**，非語音主幹。設計原則同 A1 喚醒層——**壞了就降級、語音照常**（獨立進程、可回退）。

## 0. 前提

- **硬體平台**：Genio 520（2×A78 + 6×A55、Mali-G57 MC2、NPU 10 TOPS、4GB RAM、Yocto Linux aarch64）
- **選定顯示**：1.28" **GC9A01** 圓形 SPI TFT，240×240、RGB565（部分模組附 **CST816S** I2C 電容觸控，單點）
- **選定氛圍燈**：**WS2812B / NeoPixel** 可定址 RGB LED Ring
- **約束**：主 runtime = Python/Pipecat；AI pipeline（Whisper/Qwen/Piper）已吃掉 ~2.6–3.1GB RAM；28 天 MVP；決賽斷網離線；顯示層須輕、低 CPU、好整合
- **三個用途**：(a) 呼吸燈/氛圍狀態；(b) 企鵝表情臉（待機/聆聽/說話/開心/思考）；(c) 極簡觸控小遊戲（電子雞餵食/摸摸、點擊反應類）

## 1. 結論摘要

**GC9A01 + WS2812B 這個組合完全可行**。因為是 SPI 小螢幕，**幾乎不碰 Mali GPU、RAM 只吃幾十 KB～百 KB**，跟緊繃的 AI pipeline 幾乎不搶資源，不會動搖「4GB 板選 Yocto」的 RAM 預算。

**真風險不在「能不能」，而在三個平台整合眉角**——所有查到的實作都來自樹莓派或通用 Linux/mainline kernel，**沒有一個是在 Genio 520 實測過的**。所以雷點都圍繞「別直接照抄樹莓派方案，且要自行處理 device tree / SPI-I2C bus / GPIO pinmux 並實機驗證」。

**顯示技術棧總冠軍 = LVGL（C/C++）**，一套涵蓋表情臉、氛圍微光暈、簡單觸控遊戲三用途。

## 2. 三個必避的雷（最重要）

### 2.1 凡是樹莓派專屬的都不能直接用
- WS2812B 的 `rpi-ws281x` 依賴 Broadcom BCM 的 PWM/PCM/SPI + DMA + VideoCore，**明文說不支援非 BCM 板**（"cannot support other boards like the NanoPi… outside the scope"）→ Genio 用不了。
- GC9A01 的 `dtoverlay=gc9a01` fbtft overlay 也是 RPi 專屬——「掛成 /dev/fb1、config.txt 加 dtoverlay=gc9a01」這條說法在查證中被 **0-3 否決**，不要依賴它。
- Python 驅動 `gc9a01-python` 能用，但要把 `RPi.GPIO` 換成 **`libgpiod`** 才能上 Genio。

### 2.2 WS2812B 要用 SPIdev 編碼波形，且需獨占一條 SPI bus
- **正解**：用 Linux kernel SPIdev，**MOSI 腳位餵 LED 鏈的 DIN**；每個資料位元展成 3～4 個 SPI 位元（3-bit：0→`0b100`、1→`0b110`，SPI ~2.4MHz；4-bit：0→`0100`、1→`0111`，SPI ~3.2MHz），色彩位元組相應膨脹為 3～4 位元組送出。靠 SPI 硬體時脈達 ns 級時序，避開排程器抖動。
- **不要 GPIO bit-bang**（切換 pin 約 1µs，時序不穩會閃爍；注意：「bit-bang 完全不可用」這條在查證中被 0-3 否決，代表「不是絕對不能，但不省心」——結論仍是走 SPIdev）。
- ⚠️ **關鍵約束**：LED 要一條 SPI、GC9A01 螢幕要另一條 SPI（高吞吐），兩者若共用同一條會搶 bus、干擾時序。

### 2.3 CST816S 觸控只有「單點 + 五種手勢」
- Linux 有 **mainline input/evdev driver**：`compatible = "hynitron,cst816s"`、I2C 位址 `0x15`、device tree 設 interrupt/reset GPIO，暴露為 `/dev/input/eventN`；約 kernel **v6.18（2025）** 併入 mainline，專為 Waveshare 1.28" 240×240 圓形 LCD 而寫。舊 Yocto BSP（常落 5.x/6.1 LTS）可用 **out-of-tree 模組**（kuzhylol/cst816x-driver）。
- 硬體僅支援**單點觸控**（ABS_X/ABS_Y 0–240、BTN_TOUCH，無 ABS_MT 多點）+ **五種晶片端手勢**：上/下/左/右滑、長按（映射到 BTN_FORWARD/BACK/LEFT/RIGHT、BTN_TOOL_TRIPLETAP）。
- → 小遊戲互動詞彙 = 「**點一下 + 方向滑 + 長按**」。電子雞餵食/摸摸、點擊反應類 OK；任何需雙指/多點的遊戲做不了。

## 3. GC9A01 驅動路徑（三條，擇一）

| 路徑 | 說明 | 建議 |
| :-- | :-- | :-- |
| **DRM/KMS `panel-mipi-dbi`** | mainline `drivers/gpu/drm/tiny/panel-mipi-dbi.c`，支援所有 MIPI DBI Type-C SPI + RGB565/666；GC9A01 正屬此類（帶 0x2A/0x2B/0x2C DCS 命令）；fbtft 的後繼者。經 device-tree + firmware `.bin` init 設定 | **首選**（拿到標準 framebuffer/KMS 介面） |
| **fbtft overlay** | 借 `fb_ili9340` 殼；但 `gc9a01-overlay` 本身 RPi 專屬、fbtft 已停止新增 driver | 不建議依賴 |
| **userspace spidev + Python** | `gc9a01-python`，純使用者空間；需把 RPi.GPIO 換 libgpiod | PoC 快、但 fps/CPU 較差 |

**三條路徑皆須自行提供 Genio 520 的 device tree、SPI bus 對應、GPIO pinmux 並實機驗證。**

### 更新率現實
240×240×16bit ≈ 921,600 bit/frame；SPI 40MHz 理論 **~43fps 全刷**，60–80MHz 更高。但 overlay 的 `fps=50` 是**上限參數非量測值**，實測沒人量過。→ **動畫一定要用 partial update（只重畫變動區）或 sprite 逐格設計，別假設能穩定全幀 50fps。**

## 4. 顯示/動畫技術棧比較

| 技術棧 | RAM 量級 | 適用 | 判定 |
| :-- | :-- | :-- | :-- |
| **LVGL（C/C++）** | 靜態 ~2kB + draw buffer（一行 480B / 1/10 螢幕 11.5kB / 全幀 115KB）≈ **≤140KB** | 表情+儀表+簡單遊戲三合一，內建動畫/widget，display-agnostic（只需一個 flush 回呼），官方 `lv_drivers/GC9A01.c` 現成 | **總冠軍** |
| **預渲染 sprite 逐格** | 依圖數，極省 CPU | 企鵝表情臉最省 CPU、28 天美術最好做 | **表情臉主力** |
| **lv_lottie（ThorVG）** | 中 | 想要向量絲滑情緒動畫；LVGL 內建 widget | 表情臉進階選項 |
| **rlottie（Samsung）** | 每幀 ARGB8888 240×240 ≈ 230KB | 可 aarch64 交叉編譯（Telegram/Tizen 表盤驗證） | **LVGL 官方已 deprecate rlottie player，改推 lv_lottie**；RAM 警告僅針對無 SPIRAM 的 MCU，4GB Genio 不受限 |
| **Rive** | 小檔案（比 Lottie 小 10–15×）、GPU-based | 向量動畫 | GPU 導向，對 SPI 小螢幕優勢不明顯 |
| Python：Pillow/luma.lcd/pygame-SDL | 中～高 | 整合最簡（同 Python） | fps/CPU 較差，PoC 可用 |
| **Flutter embedded / Chromium kiosk** | 重 | — | **對 240×240 SPI 小螢幕過重，不建議** |

## 5. 整合架構

顯示+燈光做成**獨立進程**，主 Python/Pipecat 只送狀態（「聆聽/說話/開心/思考」）：

- 顯示層用 **Python**（spidev + Pillow/luma）→ 跟主程式用內建 **`multiprocessing`**（Queue/共享狀態），整合最簡，fps/CPU 較差。
- 顯示層用 **C/C++ LVGL**（RAM 最省、動畫最順）→ 用 **Unix socket / pipe / shared memory + pyZMQ PUB/SUB** 送狀態。
- 好處：顯示層崩了不影響語音主幹，符合「小而可回退」。

> 註：Python 進程間 IPC 用內建 `multiprocessing` 即可，`sysv_ipc` 是給「與非 Python 程式通訊」用的。

## 6. 上板前的 Gating Items（實機必先確認）

1. **Genio 520 Yocto BSP 的 kernel 版本？** ≥6.18 可直接用 mainline CST816X driver；否則要整合 out-of-tree 模組 + 自寫 device tree。
2. **有幾條可獨立使用的 SPI controller？**（WS2812B 要一條、GC9A01 要一條）——**最關鍵硬體確認項**。不夠就外掛一顆 **RP2040/ESP32** 專門 offload 燈+螢幕，主板用 UART/I2C 下指令。
3. **LVGL(C) vs Python(spidev) 二選一**：上板第一天各跑一個最小 PoC benchmark，用實測 fps 與 CPU% 決定。
4. **表情臉：lv_lottie 向量 vs 預渲染 sprite** 的實測更新率、CPU、美術成本權衡（28 天 MVP 傾向 sprite）。

## 7. 待確認 / 侷限（Caveats）

1. **無任一來源在 Genio 520 實測**——證據皆來自 RPi 或通用 Linux/mainline。三大子系統（GC9A01 的 panel-mipi-dbi、WS2812B 的 SPIdev、CST816S 的 input driver）理論上與 SoC 無關（走 mainline DRM/spidev/i2c），但 device tree / bus 對應 / pinmux 需自行設定並實機驗證。
2. CST816S mainline driver 約 v6.18（2025）才併入；舊 BSP 須 out-of-tree 模組。
3. `fps=50` 是上限非量測；全刷實際 fps 未被實測 → 以 partial update / sprite 為前提設計。
4. WS2812B SPI 編碼需獨占一條 SPI bus 且時脈穩定；若 GC9A01 也在 SPI，須確認有兩條 SPI controller，否則搶 bus。
