# 即時語音管線／編排框架二開評估——「說說學伴」邊緣即時層骨架選型

研究日期：2026-07-04（資料查證時間點，所有活躍度數據以此為準）
研究範圍：pipeline-framework（邊緣即時層骨架：麥克風→VAD→ASR→LLM→TTS→喇叭）

## 0. 專案基準回顧

- 硬體：MediaTek Genio 520（2×A78+6×A55、NPU ~9 TOPS、**4GB RAM**、Yocto Linux aarch64）
- NPU 只能走 NeuroPilot Public（TFLite + Neuron Delegate）；LLM 走 CPU（llama.cpp，Qwen2.5-1.5B）
- 場景：**push-to-talk（手動觸發）、半雙工、單裝置、單使用者**的玩偶，非會議室/多方通話
- 延遲目標：TTS 首音 <300ms、單輪 <2s、端到端 <1.5s
- 決賽現場**斷網**、28 天工期、現行已有 FastAPI + faster-whisper + llama.cpp(Qwen2.5-1.5B) + Piper 的自建原型

## 1. 各候選逐一評估

### 1.1 Pipecat（pipecat-ai/pipecat）

- **定位**：由 WebRTC 基礎設施公司 Daily 開發的即時語音/多模態對話 AI 框架，2024/5 首發，是目前最主流的「語音 agent 編排框架」，生態環繞 Frame/FrameProcessor/Pipeline/Task 的非同步物件模型。
- **授權**：BSD-2-Clause（寬鬆）。[GitHub](https://github.com/pipecat-ai/pipecat)
- **活躍度**：13.2k stars、v1.4.0（2026-06-17 發布，累計 113 個 release）、最後 push 2026-07-03，**非常活躍**，但也代表 API 仍在快速變動中，二開時容易被上游 breaking change 拖累。
- **離線/ARM**：支援本地 Whisper（透過 faster-whisper，CPU/CUDA）、Piper TTS、Silero VAD 皆可完全離線跑（[Whisper 文件](https://docs.pipecat.ai/server/services/stt/whisper)）；也有第三方案例把 Ollama + Pipecat 跑成純本地 pipeline（[WebRTC.ventures 文章](https://webrtc.ventures/2025/03/on-premise-voice-ai-creating-local-agents-with-llama-ollama-and-pipecat/)）。但官方文件**未見針對 embedded ARM Linux（Yocto/aarch64 SBC）的部署指引**，社群案例多集中在 macOS/x86 或雲端 WebRTC transport；`LocalAudioTransport` 這條「非 WebRTC」路徑相對是社群較少走、issue 中仍在討論範例的路（[Issue #197](https://github.com/pipecat-ai/pipecat/issues/197)）。
- **打斷/VAD/半雙工**：核心賣點是 `InterruptionFrame` 廣播機制——使用者開口即可取消下游任務、停止 TTS 播放，並有 Smart Turn 語意斷句模型（~65ms CPU 推論）判斷「使用者是否真的講完了」。這一整套是為**全雙工、可被打斷**的對話設計的。官方雖有 [push-to-talk 範例](https://github.com/pipecat-ai/pipecat-examples/tree/main/push-to-talk)，但那是「用手動觸發取代 VAD/turn-detection」，等於**繞開了 Pipecat 最複雜、最有價值的那部分**（Smart Turn + barge-in），只剩下 Frame/Pipeline 的骨架在用。同時，Local audio transport 路徑上有已知的半雙工/音訊閘控 bug（[Issue #2941](https://github.com/pipecat-ai/pipecat/issues/2941)：BotStoppedSpeaking 後音訊閘保持關閉，需使用者再講話才會重置）與 `AudioBufferProcessor` 事件不觸發的問題（[Issue #4543](https://github.com/pipecat-ai/pipecat/issues/4543)），顯示非雲端 WebRTC 的路徑測試覆蓋度不如主線成熟。
- **二開難易**：Python 為主（98.4%），要接自訂 ASR（NeuroPilot TFLite）/自訂 LLM（llama.cpp CPU）/自訂 TTS，都要包成自訂 `FrameProcessor`/`Service`，且要理解整套 Frame 佇列（SystemFrame 立即處理、DataFrame/ControlFrame 排隊）與 Pipeline/Task 生命週期。對於「push-to-talk 半雙工」場景，這層抽象大多是「要繞過的東西」而非「省下的東西」。
- **結論：淘汰（作為骨架採用）**。核心價值（barge-in 打斷、全雙工、多 transport）在單裝置單使用者、手動觸發的玩偶場景幾乎用不上；剩下的 Frame/Pipeline 抽象對已有的三段式 pipeline 沒有顯著加速，反而多背一層版本快速迭代的風險與非核心路徑（LocalAudioTransport）的坑。可留作**未來雲端/雙向全雙工功能演進時的技術參考**（其 Smart Turn 斷句設計、Interruption Frame 機制值得借鏡實作思路）。

### 1.2 LiveKit Agents（livekit/agents）

- **定位**：LiveKit（WebRTC 基礎設施公司）推出的語音/視訊 agent 框架，架構前提是「一個 LiveKit Room/SFU 伺服器 + 多個 participant（人類/agent）透過 WebRTC 連進來對話」。
- **授權**：Apache-2.0（agents 主框架）；但 turn detection 模型另受 **LiveKit Model License** 限制（非純 Apache-2.0，需另外檢視條款）。[GitHub](https://github.com/livekit/agents)
- **活躍度**：11.2k stars、livekit-agents@1.6.4（2026-06-24 發布，累計 362 個 release）、最後 push 2026-07-03，非常活躍。
- **離線/ARM**：完全開源、可自架 LiveKit server（Apache-2.0），不強制依賴 LiveKit Cloud（[Self-hosting 文件](https://docs.livekit.io/transport/self-hosting/)）。但這代表**即使自架，架構上仍需要多跑一個 WebRTC SFU 伺服器**作為 agent 與音訊裝置之間的傳輸層——這是為「多方/跨網路」場景設計的必要基礎設施，對單機、麥克風喇叭直接接在同一顆 SoC 上的玩偶來說是**完全不需要的一層**。官方文件未見針對嵌入式 ARM Linux 單機部署的指引。
- **打斷/半雙工**：其半雙工/全雙工能力依賴 turn detection 模型（受額外授權限制），且與 WebRTC room 概念綁定。
- **二開難易**：要先理解 LiveKit Room/Track/Participant 概念，再理解 Agents SDK，對於「不需要網路傳輸、單裝置單使用者」的產品是本末倒置。
- **結論：淘汰**。它解決的是「agent 與遠端使用者透過網路即時通話」的問題，而說說學伴是「同一顆 SoC 上麥克風直接接喇叭」，多一層 WebRTC SFU 只會增加部署複雜度、記憶體佔用（4GB 預算緊繃）與決賽斷網風險，沒有對應的收益。

### 1.3 TEN Framework（TEN-framework/ten-framework）

- **定位**：多語言（C/C++/Go/Python/JS/TS/Rust）的即時多模態對話 AI 框架，號稱雲邊協同、支援 workflow builder，並提供 ESP32-S3 等硬體整合案例。
- **授權**：主框架標示 Apache-2.0，但 GitHub 授權偵測顯示為 `Other/NOASSERTION`（[repo](https://github.com/TEN-framework/ten-framework)），代表核心 Apache-2.0 之外**疊加了額外限制條款**，需要法務/團隊仔細審閱後才能放心商用，這在 28 天黑客松時間壓力下是額外風險。
- **活躍度**：10.8k stars、v0.11.67（2026-06-27 發布，117 個 release、2,003 commits）、最後 push 2026-07-03，活躍。
- **離線/ARM**：官方文件展示的整合案例以 OpenAI API、Deepgram ASR、ElevenLabs TTS 等**雲端服務**為主，未見明確的「純本地離線模型」部署指引；ARM 相關資訊僅見於 ESP32-S3 開發板整合與 CI workflow 的 ARM64 建置調整，**不等於官方保證 embedded ARM Linux（Yocto）上的完整 pipeline 可離線運行**。
- **二開難易**：程式碼橫跨 Python(33%)/C(22%)/C++(15%)/TypeScript(13%)/Rust(11%)/Go(3%) 六種語言混合，多語言互操作本身就是額外的建置/除錯成本；28 天內要吃透這種多語言架構、又要確認離線可行性，風險偏高。
- **結論：淘汰**。授權條款不夠乾淨、離線本地能力未經證實、多語言混合架構的學習曲線與整合成本，在時間緊迫的黑客松中都是扣分項；即使技術上可行，也不是「二開最省力」的選擇。

### 1.4 HuggingFace speech-to-speech（huggingface/speech-to-speech）

- **定位**：HuggingFace 官方的開源本地語音 agent 參考實作，`VAD → STT → LLM → TTS` 四段級聯（cascade）設計，透過 OpenAI Realtime 相容 WebSocket API 對外暴露，**每個元件都可替換**。
- **授權**：Apache-2.0。[GitHub](https://github.com/huggingface/speech-to-speech)
- **活躍度**：5.2k stars、v0.2.10（2026-06-11 發布）、最後 push 2026-07-03，活躍。
- **離線/ARM**：明確支援 ARM64（repo 內含 `Dockerfile.arm64`），且**已在生產環境跑在 Reachy Mini 機器人上**——Reachy Mini Wireless 版本搭載的是 Raspberry Pi 5（ARM Cortex-A76 四核）或更早的 Raspberry Pi 4 CM4（4GB RAM），是與 Genio 520 同量級的嵌入式 ARM Linux 單機裝置，具備麥克風陣列與喇叭（[HF 部落格](https://huggingface.co/blog/reachy-mini)、[Reachy Mini 硬體文件](https://huggingface.co/docs/reachy_mini/platforms/reachy_mini/hardware)）。這是本次評估中**唯一有「同量級嵌入式 ARM 單機、真實產品驗證」實績**的候選。本地推理透過 transformers/mlx-lm/llama.cpp/vLLM 多後端支援，完全可離線。
- **打斷/半雙工**：屬於單使用者單裝置的級聯架構，設計思路與說說學伴的邊緣即時層高度接近，比起 Pipecat/LiveKit 的多方/雲端假設更貼合。
- **二開難易**：99.9% Python，模組結構單純（`src/speech_to_speech/`、`scripts/`、`tests/`），四段式級聯直接對應現行原型的 STT/LLM/TTS 架構，改動成本低、可讀性高。
- **結論：備選（架構參考優先於整包導入）**。不建議整包依賴其 WebSocket/OpenAI Realtime 相容層與既有的元件抽象（會引入不必要的相容性包袱與依賴），但其**級聯架構設計、ARM 部署 Dockerfile、以及在 Reachy Mini 上驗證過的本地多後端推理模式**，是最值得直接拿來對照、抄工程細節的參考實作。適合在自建管線遇到「串流分段、barge-in 邊界情境」等細節問題時查閱其原始碼作為 debug 參考。

### 1.5 vocode-core（vocodedev/vocode-core）

- **定位**：早期的開源語音 LLM agent 框架，主打模組化、電話/Zoom 等場景整合。
- **授權**：MIT。
- **活躍度**：**3.8k stars，但最後一次 push 是 2024-11-15**（透過 GitHub API 直接查證：`pushed_at: 2024-11-15T22:16:58Z`），距今（2026-07）已超過 **19 個月無任何 commit**，open issues 僅 2 個（通常代表社群已不再回報，而非沒問題）。
- **結論：淘汰**。實質上已停止維護，不應作為 28 天黑客松的技術地基，即使 API 設計曾經合理，缺乏維護意味著任何 ARM/新硬體/新模型相容性問題都要自己全部承擔，風險與自建無異但少了主導權。

### 1.6 sherpa-onnx（k2-fsa/sherpa-onnx）

- **定位**：**不是編排框架，而是語音元件全家桶函式庫**——VAD、ASR（串流/非串流）、TTS、KWS（關鍵字喚醒）、speaker diarization 等，C++ 核心，官方提供 12 種語言的 binding（C++/C/Python/Go/C#/Java/Kotlin/Swift/Rust/Dart/Pascal/JS）。
- **授權**：Apache-2.0。[GitHub](https://github.com/k2-fsa/sherpa-onnx)
- **活躍度**：13.4k stars、v1.13.3（2026-06-15 發布，1,947 commits）、最後 push 2026-07-03，非常活躍，本次評估中活躍度與生態成熟度最高的專案之一。
- **離線/ARM**：**官方平台支援表明確列出 aarch64/arm32/riscv64**，是本次候選中對嵌入式 Linux 支援最直接、最無需自行摸索的一個。同時支援多種 NPU 加速：Rockchip NPU (RKNN)、Qualcomm NPU (QNN)、Ascend NPU、Axera NPU。**需注意：這裡沒有 MediaTek/NeuroPilot 的 NPU delegate**，Genio 520 上跑 sherpa-onnx 的模型仍只能透過 ONNX Runtime 走 ARM CPU，無法直接借用 sherpa-onnx 官方的 NPU 加速路徑（NPU 加速要走專案既定的 NeuroPilot TFLite + Neuron Delegate）；因此 sherpa-onnx 在本專案中的定位是「CPU 上的輕量高效元件庫」，而非 NPU 加速方案。
- **中英夾雜支援**：非常強——Zipformer/Paraformer/SenseVoice/TeleSpeech 等多個中文/方言 ASR 模型、Matcha/ZipVoice 等中文 TTS 模型，以及大量中英混合訓練的模型可選，是本次評估中對 zh-TW/中英 code-switching 場景**現成模型選擇最豐富**的一個。
- **二開難易**：C++ 核心配 Python API，屬於「函式庫」而非「框架」，不會替你決定 pipeline 的控制流——這代表它**可以無痛嵌入自建的 asyncio 狀態機**（換掉 faster-whisper 或 Piper 中的某一段），但**不能取代狀態機本身**，仍需要自己寫 push-to-talk 的生命週期管理。
- **結論：備選（元件庫定位，非骨架）**。不與「要不要用 Pipecat 類框架」這個問題直接競爭，而是「未來要不要把 faster-whisper/Piper 換成 sherpa-onnx 的 ASR/TTS 模型以優化延遲或中英夾雜準確度」的獨立決策。建議列為 P2 優化選項：若現行 faster-whisper（Whisper 系列，在中英夾雜與 ARM CPU 延遲上未必最優）在後續測試中延遲或準確度不理想，sherpa-onnx 的 SenseVoice（中英混合 ASR）與其輕量 TTS 是現成的、有官方 aarch64 支援的替代品，且换成本低（Apache-2.0，C++ 函式庫式接入不綁架構）。

### 1.7 自建薄管線基準線（asyncio + FastAPI + 狀態機，現行路線）

- **定位**：專案現行的自建路線——FastAPI 承載服務、asyncio 驅動非同步 I/O、自訂狀態機管理 push-to-talk 生命週期（IDLE/LISTENING/PROCESSING/SPEAKING/ERROR），直接串接 faster-whisper（未來可换 NeuroPilot TFLite ASR）、llama.cpp（Qwen2.5-1.5B）、Piper。
- **優勢**：
  - **零額外框架依賴**，不受任何第三方框架版本迭代/breaking change 影響（Pipecat 目前 113 個 release 顯示 API 仍快速變動）。
  - **完全掌控 push-to-talk 半雙工的狀態轉換**，不需要繞過任何為全雙工/barge-in 設計的機制（不像 Pipecat 要用 push-to-talk 範例去「關掉」Smart Turn/Interruption）。
  - **直接對接 Genio NPU pipeline**（NeuroPilot TFLite + Neuron Delegate），不需要在框架的 Service/Processor 抽象層裡额外包一層 adapter。
  - 程式碼規模可控（估計數百行等級的狀態機+膠水程式碼），除錯路徑短，符合團隊已有的 FastAPI 技術棧慣性。
- **風險/待補**：
  - 沒有現成的 barge-in、串流 partial ASR 結果處理、TTS chunked playback 最佳實踐可抄，需要自己實作並踩坑（尤其是要達到 TTS 首音 <300ms 這種對串流分段要求高的指標）。
  - 沒有現成的多裝置/多 transport 抽象，但**這正是本專案不需要的東西**，不是缺點。
  - 緩解方式：可直接參考 HuggingFace speech-to-speech 的 cascade 實作細節（尤其是它在 Reachy Mini 這類同量級 ARM 裝置上的串流/分段程式碼）與 sherpa-onnx 的 VAD/streaming ASR 範例程式碼作為「抄工程細節」的來源，而不必整包引入其依賴。
- **結論：採用（維持並強化現行路線）**。

## 2. 核心判斷題：Pipecat 的打斷/全雙工/多傳輸價值，在 push-to-talk 單機玩偶上還剩多少？

**結論：剩得不多，且拿到的部分抵不過背負的框架重量。**

推理鏈：

1. **push-to-talk 本身就是對「打斷/全雙工/turn-detection」問題的架構性迴避**——使用者用手動觸發告訴系統「我現在要說話」，系統不需要用 VAD + 語意斷句去猜測「使用者是否講完了」，也不需要在使用者開口時搶佔性地打斷正在播放的 TTS（因為半雙工設計下，使用者要先讓系統停止說話、再按下觸發）。Pipecat 的招牌能力——`InterruptionFrame` 廣播取消、Smart Turn ~65ms 語意斷句——正是為了解決 push-to-talk 迴避掉的那個問題而存在的。官方自己的 push-to-talk 範例，本質上就是「把 Pipecat 最有價值的那塊功能關掉，只留下 Frame/Pipeline 骨架」。
2. **多傳輸（WebRTC/Daily/LiveKit/電話）在單裝置單使用者的玩偶上完全用不到**——聲音是本地麥克風直接進、本地喇叭直接出，不涉及跨網路傳輸、不涉及多方併發。Pipecat/LiveKit 生態中最成熟、測試覆蓋最好的路徑（雲端 WebRTC transport）用不上；用得上的路徑（`LocalAudioTransport`）反而是社群案例較少、已知有半雙工閘控 bug 的相對次要路徑。
3. **框架帶來的依賴重量是實質成本，不是理論成本**——4GB RAM 預算緊繃，Yocto aarch64 是相對小眾的部署環境，Pipecat 的 API 一年內發了 113 個 release（代表還在快速演進），28 天內對接自訂 NeuroPilot TFLite ASR/自訂 llama.cpp LLM service，都要自己包 FrameProcessor/Service adapter——這份整合工作量，自建狀態機也要做（串接 ASR/LLM/TTS），但自建不需要**額外**學習 Frame 佇列語意、Pipeline/Task 生命週期、以及在框架升級時重新驗證相容性。
4. 因此，在「28 天內哪個風險低」這題上：**自建 300 行等級狀態機的風險低於引入 Pipecat**。真正該花力氣借鏡的，是 Pipecat 的 Interruption Frame 設計思路（作為未來要做全雙工/可打斷功能時的參考架構）與 HuggingFace speech-to-speech 這種形狀更貼近本專案（單機、級聯、已驗證 ARM 嵌入式）的參考實作。

## 3. 排序推薦表

| 排名 | 候選 | 授權 | 活躍度(最後push) | ARM/離線 | 建議定位 |
|---|---|---|---|---|---|

## 4. 28 天內的整合路徑（邊緣即時層）

- **Day 1–3**：定案自建狀態機骨架——明確定義事件與狀態轉換（IDLE → LISTENING → PROCESSING → SPEAKING → ERROR/兜底），把現行 FastAPI+faster-whisper+llama.cpp+Piper 三段串起來，先求「能跑通一輪」。
- **Day 4–7**：實作串流化——ASR partial result 處理、LLM 逐 token 流式輸出接 TTS 分段合成、TTS chunked playback，目標打穿 TTS 首音 <300ms。此階段參考 HuggingFace speech-to-speech 的 cascade 串流程式碼、sherpa-onnx 的 streaming ASR/VAD 範例作實作對照（不引入其依賴，只抄工程做法）。
- **Day 8–10**：對接 Genio NPU——把 ASR（或部分模型）遷到 NeuroPilot TFLite + Neuron Delegate，實測延遲與記憶體佔用，確認 4GB RAM 預算下的可用餘量（目標 OS+runtime 後留 2.5–3GB 給模型）。
- **Day 11–14**：中英夾雜（code-switching）調校——若 faster-whisper 在 zh-TW/中英混合上準確度或延遲不理想，評估替換為 sherpa-onnx 的 SenseVoice/Zipformer 雙語模型（P2 決策點，此時應該已有實測數據支撐）；同步調 LLM 鷹架 prompt 與 TTS 語者/語言切換。
- **Day 15–18**：兜底話術、錯誤處理、離線容錯路徑——確保 ASR/LLM/TTS 任一環節逾時或失敗時有優雅降級與固定話術，為決賽斷網 demo 做保底。
- **Day 19–21**：本地 SQLite 記錄整合、與雲端助教層/教師儀表板的資料交換介面對接（若邊緣層需要回傳事件）。
- **Day 22–24**：端到端延遲與記憶體壓力測試——量測單輪 <2s、端到端 <1.5s 目標是否達成，在真實 Genio 520 硬體上做記憶體 profiling，抓 OOM 風險。
- **Day 25–26**：兒少內容安全檢查、決賽斷網 demo 全流程排練（含異常情境：無網路、麥克風雜訊、長句中英夾雜）。
- **Day 27–28**：Buffer——修復排練中發現的問題、決賽最終排練。
