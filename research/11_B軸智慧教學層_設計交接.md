> 母脈絡：先讀 `research/10_雙網混成語音方案總分析.md`（選型已定案）。本檔只談 B 軸。

## 0. 為什麼可以平行開 session 做 B 軸（安全性）

專案切成兩條大軸，介面乾淨、可獨立設計/實作/demo：

- **A 軸　互動/傳輸層**：管「怎麼聽、怎麼說」——雙網混成、喚醒、全雙工 barge-in。（另一 session / 既有工作串）
- **B 軸　智慧/教學層**：管「說什麼、教什麼」——雙 Agent 閉環、長期記憶、CEFR 階梯、Guardrails。（本檔）

**交界只有一個點**：`pipeline._process_text()` 裡的「一輪 text in → reply text out」。A 軸不改這條線的語意，B 軸不碰音訊傳輸。兩邊各自演進，最後在 pipeline 對接即可。所以平行開 session **不會衝突**。

## 1. 雙 Agent 願景 vs 現有 code（精準對照，已驗證 file:line）

| 願景模組 | 現有對應 | 狀態 / 缺口 |
|---|---|---|
| **陪聊 Agent**（即時、人設、引導開口） | `server/scaffold.py`（規則鷹架）+ `server/llm.py` `EdgeLLM.generate` | 已有，但 `llm.py:44 _SYSTEM_PROMPT` **完全靜態**、無人設變化、不吃學生狀態 |
| **導師 Agent**（異步、評估、產策略） | `server/diagnose.py` `generate_diagnosis()`（`diagnose.py:427`） | **已存在！** 讀近 10 筆 → 四維分數 + strengths/weaknesses + emotional_status + instructions。**但 instructions 三欄是 classroom/device/peer（給老師），沒有回寫給陪聊 Agent** |
| 記憶庫（短期） | `pipeline._process_text` 單輪 context | 有（單輪） |
| 記憶庫（長期 profile） | `store` interactions 表（SQLite） | 有原始資料，**未抽成「學生模型／興趣／已掌握字彙／歷史錯點」** |
| 難易度階梯 / 主題迭代 | 無 | **全新**（diagnose 只有四維 0–100，無 level 概念與升降級規則） |
| Guardrails / 隱私 | `scaffold.safety_check`（禁詞） | 禁詞已有；COPPA、資料留存、家長同意屬橫切，未系統化 |

**最大洞察**：所謂「自動迭代的精髓」＝導師分析完、回寫陪聊的 system prompt。現在就差把 `diagnose.py` 的輸出接回 `llm.py` 的 prompt 這一條線。

## 2. B 軸子系統拆解（建議一次只設計一個）

- **B1　雙 Agent 閉環**：導師（diagnose）評估 → 產「陪聊下一輪策略指令」→ 回寫進 `llm.generate` 的 prompt。
- **B2　長期記憶庫**：從 interactions 抽學生 profile（興趣、已掌握字彙、歷史錯點），供導師與陪聊取用。
- **B3　CEFR 難易度階梯 + 主題迭代**：把四維分數映射到 Level 微階梯（如 1-1～5-5），定升/降級與換主題規則。
- **B4　Guardrails / 隱私（COPPA，橫切）**：不當內容防護、語音資料留存政策、家長同意。

**建議排序理由**：B1 風險最低、離 demo 最近（diagnose 已在，只要「回寫 prompt」就能秀出「玩偶會自己調難度」＝自動迭代最小可行展示）。B2 是 B1/B3 的資料底座，緊接第二。B3 需要 B2 的字彙掌握資料才紮實。B4 全程橫切、隨時補。

## 3. B1 最小可行閉環（第一個要設計的子專案）

**目標展示**：連續幾輪流暢 → 玩偶自動升句型；卡住/沉默 → 玩偶自動降成二選一。**不引入 LangGraph/AutoGen**（見 §4）。

三個小改動（皆低侵入，維持現有降級韌性）：

1. **導師端**：`diagnose.generate_diagnosis` 增產一個「陪聊指令」欄位（例：`companion_directive`），內容如
   > 「學生 Level 2，能用單字/短句描述動物。下一步：引導使用顏色形容詞。把話題帶到『農場動物』，多問 What color is the pig? 若困惑則退回單字引導。」
   規則式先給（依現有 flags/scores 組），有 `ANTHROPIC_API_KEY` 時走 Sonnet 生成、失敗 fallback（沿用現有雙軌 pattern）。
2. **陪聊端**：`EdgeLLM.generate` 增收一個 `directive: str | None` 參數，拼進 `user_prompt`（`llm.py:114`）；雲端 provider 同理。`_SYSTEM_PROMPT` 可保留骨架、把可變策略放 user turn。
3. **接線**：`pipeline._process_text`（`pipeline.py:179`）在呼叫 `llm.generate` 前，讀「最新 directive」傳入。directive 來源：先放記憶體/`store`（每 N 輪或每 session 結束由導師更新一次，**異步**，不擋即時對話）。

**注意韌性**：directive 為 None 時行為＝現況，一切照舊降級。導師是異步的，絕不進即時路徑。

## 4. 誠實對照：貼文通用建議 vs 本專案已實測裁決（別照抄）

1. **STT 用 Whisper** → bench 實測 **SenseVoice 繁中 CER 2.8% vs whisper 6.8%、快 4×**。維持 SenseVoice，whisper 只留 fallback。
2. **TTS 用 ElevenLabs/Azure** → 雲端付費、離線就沒了。選 **sherpa-onnx**（Apache、本地離線）；童聲/情緒用 **Kokoro**（已掃 22 個 sid 待人耳選）補，不上雲。
3. **雙 Agent 用 LangGraph/AutoGen** → 導師半套用純 `urllib`+Anthropic 就做到了（見 `diagnose.py`），**不需 agent 框架**。真正值得引框架的是 A 軸全雙工語音（Pipecat 類），不是 agent 編排。**別把兩件事混在一起選框架。**

（硬體：MediaTek Genio 520，現為 PC/瀏覽器原型，軟體先行。）
