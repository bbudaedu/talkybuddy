# 說說學伴 TalkyBuddy — 雙網混成語音方案總分析

> 日期：2026-07-07
> 整合來源：research/07（框架選型）、research/08（實作規劃）、talkybuddy/bench/（本機實測）
> 需求：雲端優先雙網混成 + 離線邊緣 + 繁中（台灣）與英文雙語 + 兒童語音；push-to-talk 半雙工單機玩偶；硬體目標 MediaTek Genio 520（現階段 PC/瀏覽器原型）

---

## 一、總結論（TL;DR）

| 決策項 | 結論 | 信心 |
|---|---|---|
| **語音編排層** | 自建薄骨架（維持現有 FastAPI + WebSocket），不引入 Pipecat/LiveKit/TEN/Vocode | 高（85%） |
| **雲端對話層** | 維持 `diagnose.py` urllib 直呼 Anthropic Messages API；不採用 Hermes Agent | 高 |
| **ASR/STT** | 維持 SenseVoice（實測中文 CER 2.8%、延遲 91ms，全面勝 whisper） | 高 |
| **TTS** | 維持 sherpa-onnx；Kokoro 列童聲增補候選（需人耳定奪） | 中高 |
| **本地 LLM** | **建議 fallback 改用 Qwen2.5-0.5B**（延遲 4.6s→1.9s，品質仍足） | 中高 |
| **架構主軸** | 雲端優先 + 斷網自動 fallback 本地 LLM（僅簡單功能） | 高 |

**一句話**：語音層自建薄骨架不動；三類元件既有選型站得住，唯一實質改進是本地 LLM 換 0.5B 把延遲砍半；最大價值來自雙網混成架構——線上用雲端強 LLM 消除延遲，斷網用輕量本地模型維持基本互動。

---

## 二、語音編排框架選型（詳見 research/07）

在「push-to-talk 半雙工 + 雲端優先 + 斷網切本地 + Genio 520 NPU」四重約束下，正面比較五條路線：

| 路線 | fit | 一句話 |
|---|---|---|
| **自建薄骨架** | **5/5** | 唯一精準契合四重約束、零授權風險、遷移成本最低 |
| Pipecat | 2/5 | 核心價值全雙工 barge-in，半雙工用不上 |
| LiveKit Agents | 2/5 | WebRTC SFU 為多方串流而生，單機殺雞用牛刀 |
| TEN Framework | 2/5 | 圖狀 dataflow 高速迭代、C++/Go 棧與 Python 阻抗不匹配 |
| Vocode | 1/5 | 電話 IVR 導向、2026 維護放緩 |

**關鍵洞察**：雲端優先/斷網 fallback 的 network-aware routing 無任何框架內建，全都要自己寫，引入框架只是徒增抽象與授權風險。雲端層 Hermes 亦不採用（仍單機助理定位、Bedrock 整合薄弱）。

## 三、雙網混成架構方向（詳見 research/08）

```
[網路狀態機] --偵測--> online / offline
     |                    |
     v (online)           v (offline)
[雲端優先路由]        [本地 fallback]
 對話+教案+鷹架+       僅簡單功能：
 學生學習管理          跟讀、說故事
  → 雲端強 LLM          → 本地輕量 LLM
        \              /
     [對話 LLM provider 抽換介面]
      CloudProvider / LocalProvider
        |
[現有 pipeline.py：ASR → scaffold → LLM → TTS]
```

- **網路狀態機**：非同步健康檢查、防抖動 hysteresis（避免高延遲誤判斷網）、保留手動 override 供 demo。
- **路由層**：功能分級（online 全功能走雲端；offline 降級為跟讀/說故事）；雲端逾時即時降級本地。
- **provider 抽換介面**：CloudProvider（Anthropic）/ LocalProvider（本地 gguf）統一介面，最小侵入現有 code。
- **前置事項**：先處理 llm.py/pipeline.py/tts.py 未 commit 的 WIP。

## 四、三類元件本機實測（詳見 talkybuddy/bench/REPORT.md）

> 方法：現有 TTS 合成 10 句中英雙語（含破音字/數字/中英夾雜），原文即 ground truth 算 ASR CER。
> 限制：合成語音 ≠ 真實兒童語音；延遲為 PC x86，非 Genio 520 NPU 實機。

### 4-1 ASR / STT

| 指標 | SenseVoice（現用） | faster-whisper small |
|---|---|---|
| 平均 CER | **2.4%** | 5.2% |
| 中文 CER | **2.8%** | 6.8% |
| 英文 CER | 3.0% | **2.0%** |
| 平均延遲 | **91 ms** | 384 ms（慢 4.2×） |

- SenseVoice 破音字全對；whisper 中文同音錯（讀一「篇」、長「的」很高）。
- **維持 SenseVoice**；whisper 英文略優但中文差+慢 4 倍，僅續作 fallback。

### 4-2 TTS

| 指標 | sherpa-onnx+Piper（現用） | Kokoro 多語 |
|---|---|---|
| 平均延遲 | **184 ms** | 705 ms（慢 3.8×） |
| 音色數 | 單一 voice | **53 speakers** |
| 授權 | Apache runtime（espeak GPL 殘留） | 同樣 espeak-ng 殘留 |

- **維持 sherpa-onnx**（延遲優）；Kokoro 音色多（利於找童聲），已掃描 sid 產 22 個 wav（`bench/out/voices/`）待人耳定奪。

### 4-3 邊緣 LLM（延遲階梯，5 模型）

| 模型 | 平均延遲 | tok/s | 品質 |
|---|---|---|---|
| **Qwen2.5-0.5B** ⭐ | **1.93 s** | ~40 | 親切、台灣用語、能帶讀+引導說故事，完全堪用 |
| Gemma-2-2b-it | 3.77 s | ~20 | 好但慢一倍、無明顯優勢 |
| Qwen2.5-1.5B（現用） | 4.56 s | 19 | 簡短到位 |
| Qwen3-1.7B | 5.12 s | 19 | 引導更豐富、支援 thinking mode |

- **重大發現：Qwen2.5-0.5B 延遲砍半（4.6s→1.9s），品質對簡單功能足夠。**
- **建議**：本地 fallback 改 Qwen2.5-0.5B（延遲關鍵）；Qwen2.5-1.5B 留作「本地較完整功能」中檔；Qwen3 續列非即時/thinking 增補；Gemma 不引入。

## 五、最終選型建議

| 類別 | 最終建議 | 增補候選 |
|---|---|---|
| 語音編排 | **自建薄骨架** | —（不引入任何框架） |
| 雲端對話 | **Anthropic Messages API 直呼** | —（不採 Hermes） |
| ASR/STT | **SenseVoice** | whisper（英文/fallback） |
| TTS | **sherpa-onnx + Piper voice** | Kokoro（童聲，需選 sid） |
| 本地 LLM | **Qwen2.5-0.5B（fallback 即時）** | Qwen2.5-1.5B（完整）、Qwen3（thinking） |
| 雲端 LLM | **claude-sonnet-5** | — |

**核心痛點與解法**：本地 LLM 延遲（4–6s）是唯一瓶頸（ASR/TTS 皆毫秒級）。雙路並進——(1) 雲端優先架構用雲端強 LLM 徹底消除延遲；(2) 本地 fallback 改 0.5B 把延遲壓到 ~2s。

## 六、限制與待驗證

1. **真實兒童語音**：本機用合成語料，無法反映真實口音/發音不標準；建議錄音補測真實 CER。
2. **Genio 520 上板**：所有延遲為 PC x86，NPU 量化/加速實機延遲須上板才知。
3. **Kokoro 童聲 A/B**：需人耳聽 `bench/out/voices/` 挑 sid，才知是否值得用 3.8× 延遲換音色。
4. **Qwen2.5-0.5B 品質底線**：對「較完整功能」（教案、鷹架）是否足夠，需再測；目前僅驗證簡單功能足夠。
5. **espeak-ng GPL 殘留**：TTS 音素化資料仍 GPL-3.0，兩方案皆未解，待找替代。
6. **research/09**：2026 web 廣度事實研究仍在背景跑，完成後可與本機實測交叉對照補強。
