# 整機 AI 語音玩具／語音助理二開專案評估——「說說學伴」邊緣即時層基底選型

研究日期：2026-07-04（資料查證時間點，所有活躍度數據以此為準，皆以 GitHub API `pushed_at`/`stargazers_count`/`license` 直接查證，非憑記憶）
研究範圍：turnkey-toys（找「已經是完整 AI 語音玩具/語音助理」的開源專案，評估整包二開 vs 自建）

## 0. 專案基準回顧

- 硬體：MediaTek Genio 520（2×A78+6×A55、NPU ~9 TOPS、**4GB RAM**、Yocto Linux aarch64），僅能用 NeuroPilot Public（TFLite + Neuron Delegate）；LLM 走 CPU（llama.cpp，Qwen2.5-1.5B）
- 場景：**push-to-talk（手動觸發）、半雙工、單裝置、單使用者**的教學玩偶，非會議室/多方通話，非通用聊天玩具
- 延遲目標：TTS 首音 <300ms、單輪 <2s、端到端 <1.5s；決賽現場**斷網**
- 現況：已有自建 PC 原型（FastAPI + faster-whisper + llama.cpp Qwen2.5-1.5B + Piper）在建置中，28 天工期
- 本題核心：找「已解決 VAD/打斷/佇列/插拔」的完整語音玩具專案，評估整包改造 vs 自建何者 28 天內完成度更高

## 1. 各候選逐一評估

### 1.1 xiaozhi-esp32 + xiaozhi-esp32-server（78/xiaozhi-esp32、xinnan-tech/xiaozhi-esp32-server）

- **定位**：中文開源社群目前最大的「AI 語音玩具」專案。`78/xiaozhi-esp32` 是 ESP32 端韌體（客戶端），`xinnan-tech/xiaozhi-esp32-server` 是配套後端服務，由華南理工大學劉希垚（Siyuan Liu）教授團隊主導，社群規模是本次評估中最大的一個。ASR/LLM/TTS 全部走 provider 插件抽象，支援雲端 API 與本地模型混搭，有記憶系統與角色（character）設定。
- **授權**：MIT（[GitHub](https://github.com/xinnan-tech/xiaozhi-esp32-server)），寬鬆、可商用、可魔改，無 GPL 傳染疑慮。
- **活躍度**：**9,970 stars**、最新 release **v0.9.5（2026-06-30）**、主分支累計 **3,652 commits**、最後 push **2026-07-01**，open issues 750（社群量體大、活躍但也代表 issue 累積量大）。本次評估中活躍度與社群規模最高。
- **架構與目錄結構**：模組化，`main/xiaozhi-server` 是**獨立可運行的 Python 核心**（VAD→ASR→LLM→TTS 線性 pipeline，可選接 intent 辨識與記憶），`manager-web`（Vue）/`manager-api`（Java）是**選配的管理後台**，官方明確提供「最簡化安裝」（僅 Server）路徑，代表**不需要吃下整個 Java/Vue 管理系統**就能跑核心語音服務。這降低了「程式碼規模」這項風險——真正要讀懂、要改的核心其實只是 Python 那一塊。
- **離線/ARM(aarch64) 支援**：
  - ASR：支援 FunASR（記憶體需求高，≥3GB）與 **sherpa-onnx-asr**（記憶體 <1GB，且 sherpa-onnx 官方明確提供 **aarch64 Python wheels**，[k2-fsa/sherpa-onnx](https://github.com/k2-fsa/sherpa-onnx)、[Issue #315](https://github.com/xinnan-tech/xiaozhi-esp32-server/issues/315)），是本次候選中對 ARM 記憶體預算最友善的組合。
  - LLM：支援 **Ollama**（本地離線，可跑 Qwen2.5 系列，[Issue #2855](https://github.com/xinnan-tech/xiaozhi-esp32-server/issues/2855)）、ChatGLMLLM 等雲端 API 皆可切換，插件化。
  - TTS：預設 EdgeTTS 為**雲端服務**（斷網會失效，不可直接用於決賽現場），但也支援本地 FishSpeech/GPT-SoVITS（[Issue #2272](https://github.com/xinnan-tech/xiaozhi-esp32-server/issues/2272)）——這兩者是重量級 GPU 導向 TTS，在 4 核 A55+2 核 A78 的 CPU-only 環境上效能未知，需自行驗證或換成 Piper（社群無現成 provider，需自己補一個 TTS provider class，工作量不大，因為介面已抽象化）。
  - **官方文件未見任何針對 Yocto/embedded ARM Linux 的部署指引**——官方 Docker 部署鎖定的是一般 Ubuntu-based x86/ARM server，**不是 Yocto image**。Yocto 通常沒有標準 pip/apt 生態，FunASR/sherpa-onnx/Ollama 這類套件要嘛需要交叉編譯，要嘛需要在 image 內塞一個完整 Python venv——這件事無論用不用 xiaozhi-esp32-server 都要自己做，xiaozhi 在這塊**沒有额外幫團隊省到力**。
  - **NPU（NeuroPilot TFLite + Neuron Delegate）完全沒有對應 provider**——這是本專案技術棧最特殊、也最需要自建的一塊，xiaozhi 生態圈沒有任何人做過 MediaTek NeuroPilot 整合（其 ARM NPU 相關討論僅止於通用 aarch64 CPU 部署），這部分無論選哪個候選都要從零寫。
- **中英夾雜(zh-TW/code-switching)支援**：介面提供簡中/繁中/英文切換；ASR/TTS 模型生態以**簡體中文/中國大陸用語**為主（FunASR、CosyVoice 等模型訓練語料多為中國大陸普通話），**未見針對台灣腔/繁體用語特化的模型或文件**，需自行評估辨識準確度是否適用台灣國小學童口音與用語（這與團隊已在測試的 faster-whisper 面臨的問題類似，並非 xiaozhi 特有優勢）。
- **延遲特性**：官方稱簡化版 2 核 4G（FunASR）即可跑，串流 ASR/TTS 設計上可支援低延遲，但**沒有針對本專案「TTS 首音 <300ms」這種嚴苛指標的公開 benchmark**，需自行實測。
- **打斷/佇列/插拔（本題最關鍵的「已解決」部分）**：協定層（`docs/websocket.md`／`websocket_zh.md`）明確定義了 hello 交握、Opus 二進位音訊幀、`type` 為核心欄位的 JSON 訊息（TTS/STT/IoT/WakeWord 等狀態管理），VAD 與打斷（barge-in）邏輯已內建於 server 端狀態機，ASR/LLM/TTS 三段都走**統一的 base class 插件介面**，新增一個 provider 只需要繼承介面、實作固定方法——這正是題目所指「已解決 VAD/打斷/佇列/插拔」的部分，**確實比從零開始省力**。
- **客戶端替換可行性（能否用板上程式取代 ESP32）**：**可行**。協定完全走標準 WebSocket + JSON + Opus 二進位幀，官方文件明確說明「協定可擴充，欄位可自行約定」，本質上是一個**與硬體無關的應用層協定**。團隊可以用 Python/C 在 Genio 520 上寫一個相容協定的 client（甚至可以把 client 和 server 都跑在同一顆 SoC 上，直接以本地 loopback WebSocket 溝通，省去真正的 ESP32），技術上沒有阻礙，但**這件事本身就是一次不小的整合工程**（要讀協定文件、實作 hello 交握、Opus 編碼、狀態機同步），不是「裝上就能跑」。
- **記憶與角色設定**：本地短期記憶 + mem0ai + PowerMem 三種可選記憶方案，角色設定走靜態 system prompt/character 設定檔——這是「通用聊天玩具」的記憶設計（讓 LLM 記得使用者說過什麼），**不是教學鷹架所需的「學習狀態追蹤」（哪些字錯過、難度該不該升級、今天教到第幾課）**，兩者概念不同，不能直接沿用。
- **二開難易**：核心 Python 服務程式碼規模適中、有插件基底類別可循，是本次評估中「架構文件最完整、插件點最明確」的一個。但要注意：repo 整體語言分布 JS 42.2%／Vue 17.7%／Python 17.3%／Java 16.5%，**代表大部分程式碼量其實在管理後台**，真正要碰的核心 Python server 相對是小部分——這是優點（不用全部讀懂）也是提醒（不要被總 commit 數/repo 大小嚇到而高估或低估工作量，實際要精讀的範圍比表面數字小）。
- **對本專案的適配結論：備選（設計參考優先於整包導入）**。授權乾淨、社群活躍、協定與插件設計成熟，是本次所有候選中**唯一具備「整包扛著跑」技術可行性**的專案；但它解決的「已完成」部分（VAD/打斷/佇列/插拔）團隊自建原型（FastAPI + asyncio 狀態機）也已經在做同等級的事，而本專案真正的差異化與時間風險——鷹架引擎、教學評量、教師端報表、兒少安全層、NeuroPilot NPU 整合——xiaozhi 完全沒有,無論選誰都要在 28 天內從零生出來。相較「花數天讀懂一個陌生的 3,652-commit 外部 codebase 再開始魔改」，直接在已跑通、團隊自己完全掌握的原型上疊加這些模組風險更低。**建議：不整包 fork，但花 1 天 code reading，具體抄以下三項資產**：①WebSocket JSON 協定 schema 與狀態機事件設計、②ASR/LLM/TTS provider 插件基底類別的介面切法、③記憶模組的資料結構設計（可以魔改成教學進度追蹤用）。

### 1.2 FoloToy / folotoy-server-self-hosting

- **定位**：商業 AI 玩具公司 FoloToy 開放的自架伺服器設定檔倉庫（不是完整原始碼倉庫），讓使用者可以自架後端取代 FoloToy 官方雲端，音訊資料透過 MQTT Broker 發布，TTS 供應商含 OpenAI TTS/Azure/ElevenLabs/Edge TTS/阿里雲 TTS/網易 EmotiVoice/OpenVoice。
- **授權**：GPL-3.0（[LICENSE](https://github.com/FoloToy/folotoy-server-self-hosting/blob/main/LICENSE)）——比 MIT/Apache 嚴格，若團隊未來要商業化或不想公開衍生程式碼，GPL-3.0 是額外法務考量。
- **活躍度**：590 stars、最後 push **2026-02-01**、最新 release 卡在 **2023-12-01**、open issues 僅 11 個。push 時間與 release 時間不一致，代表近期雖有零星 commit 但**沒有正式版本推進**，社群量體遠小於 xiaozhi（590 vs 9,970 stars），且是「設定檔倉庫」而非完整應用程式碼，**不是一個可以整包拿來跑的完整系統**，更接近「設定範例」。
- **離線/ARM**：Docker Compose 可本地部署，但 TTS/ASR 選項多數為**雲端 API**（OpenAI/Azure/ElevenLabs/阿里雲），本地選項（Edge TTS 其實也是雲端、EmotiVoice、OpenVoice）較少且未見 ARM/aarch64 專項驗證。
- **中英夾雜支援**：README 有簡中版本，但整體社群與案例以英文/國際市場為主。
- **對本專案的適配結論：淘汰**。社群量體小、非完整原始碼（只是自架設定）、以雲端 API 為主要設計前提（與決賽斷網需求衝突）、GPL-3.0 授權比 xiaozhi 的 MIT 多一層限制，三者疊加下沒有比 xiaozhi-esp32-server 更值得投入的理由，也沒有值得抄的獨特架構亮點。

### 1.3 OpenVoiceOS（ovos-core）／Neon AI（NeonCore）

- **定位**：Mycroft 停止維運後由社群接手的開源語音助理平台，OVOS 是核心框架、Neon AI 是其商業/應用層延伸，兩者共用大量元件，架構圍繞「intent skill」模型（類似 Alexa Skill 的技能式架構），設計目標是通用桌面/家庭語音助理而非玩具。
- **授權**：兩者皆 Apache-2.0（[ovos-core](https://github.com/OpenVoiceOS/ovos-core)、[NeonCore](https://github.com/NeonGeckoCom/NeonCore) 顯示為 NOASSERTION 但主體採 Apache 條款，仍建議實際引用前再次確認 repo 內 LICENSE 檔案）。
- **活躍度**：ovos-core 278 stars、最後 push **2026-07-03**（非常新鮮）；NeonCore 205 stars、最後 push 2026-07-02。**push 頻率很高但 star 數遠低於 xiaozhi（不到 xiaozhi 的 3%）**，反映的是一個技術上持續有人維護、但社群規模與案例量都小很多的小眾生態。
- **離線/ARM**：Python 為主，理論上可跨平台，社群案例中有樹莓派部署經驗，但**没有針對 8 核異構 SoC + NPU 的優化案例**，且其核心「intent skill」架構是為「你說一句話，系統匹配意圖後執行對應技能（開燈、查天氣）」設計，**與「LLM 生成式教學鷹架對話」的生成式、多輪追問場景是兩種不同的互動模型**，要拿來做教學鷹架需要繞過而非利用其 skill 匹配機制。
- **對本專案的適配結論：淘汰**。技術上活躍、授權乾淨，但（1）架構前提（intent-skill 匹配）與教學鷹架所需的生成式多輪對話設計理念不合，硬套等於捨棄其核心價值只留下殼；（2）社群規模遠小於 xiaozhi，遇到問題時可參考的案例與文件少很多；（3）沒有比 xiaozhi 更突出的「玩具化」現成度。可作為「開源語音助理如何做 skill/intent 抽象」的次要架構參考，但不建議投入時間整合。

### 1.4 Rhasspy 3 / Wyoming 協定生態（wyoming-faster-whisper/wyoming-piper/wyoming-satellite）

- **定位**：Wyoming 是 Rhasspy 團隊設計的輕量語音元件通訊協定（stdin/stdout JSON + binary，或走 TCP），讓 ASR/TTS/wake-word 等元件可以獨立成「衛星」服務被 Home Assistant 等中控呼叫，`wyoming-faster-whisper`／`wyoming-piper` 是把 faster-whisper／Piper 包裝成 Wyoming 協定伺服器的 wrapper。
- **授權**：全部 MIT。
- **活躍度與重大變化**：
  - `rhasspy/rhasspy3`（整合工具箱本體）**已封存（archived），最後 push 2023-12-26**，是死專案，不應作為工具箱依賴。
  - `wyoming-faster-whisper`：349 stars、最後 push **2026-07-01**，仍活躍維護，可作為「把團隊現有 faster-whisper 包一層 Wyoming 協定」的輕量參考（但團隊目前架構是直接呼叫 faster-whisper函式庫，不需要多一層協定）。
  - `wyoming-piper`：183 stars、最後 push 2026-04-07，仍維護中。
  - **`wyoming-satellite`（衛星端 orchestrator）已於 2026-01-27 被官方封存**，原因是「已被 **Linux Voice Assistant**（走 ESPHome 協定而非 Wyoming）取代，因其支援媒體播放器、停止喚醒詞、啟動/延續對話、計時器等新功能」（[GitHub 說明](https://github.com/rhasspy/wyoming-satellite)）。**這代表 Wyoming 協定生態的「衛星端」技術路線正在被 Home Assistant 官方（現由 Open Home Foundation 主導）放棄，轉往 ESPHome 協定**——對一個要在 28 天內選定長期技術路線的專案而言，這是明確的**趨勢警訊**：不建議現在才選一個「原廠正在淘汰」的協定當地基。
- **繼承者 `OHF-Voice/linux-voice-assistant`**：Apache-2.0、509 stars、最後 push **2026-07-03**（極活躍）、**官方明確支援 linux/aarch64**（含 amd64），最低硬體需求僅 1GHz CPU/512MB RAM，遠低於 Genio 520 規格；本地喚醒詞用 OpenWakeWord 或 MicroWakeWord。但**這個專案的定位是「Home Assistant 的語音衛星端」，設計前提是背後一定要有一個 Home Assistant 實例負責對話/LLM pipeline 的決策**，本身不含 LLM 教學鷹架邏輯，且與 HA 生態系（entity/automation/conversation agent）深度綁定，要抽出來單獨用等於要重寫掉它與 HA 溝通的那一層，只留下「喚醒詞偵測 + 音訊擷取」這一小塊價值。
- **對本專案的適配結論：淘汰（工具箱/衛星整包），wyoming-faster-whisper 與 linux-voice-assistant 的喚醒詞模組列為窄範圍參考**。Wyoming 協定本身正處於官方路線轉換期（衛星端已被放棄），不適合作為新專案的長期依賴；`linux-voice-assistant` 技術上 ARM64 現成可用，但綁死 Home Assistant 中控架構，與「單機獨立運作、斷網可用」的玩偶需求不合，需要拆掉的部分比能沿用的部分多。唯一值得參考的是其 **OpenWakeWord/MicroWakeWord 本地喚醒詞方案**——如果團隊未來要從 push-to-talk 升級到真正的喚醒詞觸發，這是現成、輕量、ARM 友善的選項可以參考其模型與整合方式。

### 1.5 Willow（HeyWillow/willow）

- **定位**：以 ESP32-S3-BOX 系列硬體為目標的開源語音助理，目標是做到與 Amazon Echo/Google Home 相當的體驗，架構分為**裝置端韌體（ESP-IDF/C，跑在 ESP32-S3 微控制器上）**與**Willow Inference Server（WIS，後端推理伺服器，可自架）**兩部分。
- **授權**：Apache-2.0（[GitHub](https://github.com/HeyWillow/willow)）。
- **活躍度**：3,063 stars、最後 push **2026-02-13**、open issues 27（相對 star 數不算多，但距今已近 5 個月無 push，維護節奏明顯放緩，**並非近期最活躍的候選**，但也未到停止維護/封存的程度）。
- **離線/ARM**：語言為 C（韌體端），**目標硬體是 ESP32-S3 微控制器，不是 embedded ARM Linux SBC**——這與 xiaozhi-esp32 的定位完全一樣，都是「MCU 端韌體 + Linux 後端服務」的兩段式架構，且**同樣不存在拿它的 MCU 韌體直接跑在 Genio 520（一顆完整跑 Yocto Linux 的 8 核 SoC，不是 MCU）上的可能性**，能借的只有後端 Willow Inference Server（WIS）這一段。
- **對本專案的適配結論：淘汰**。與 xiaozhi 相比，Willow 的裝置端韌體對本專案完全用不上（Genio 520 是跑完整 Linux 的 SoC，不是 ESP32-S3 那類微控制器），後端 WIS 的功能（Whisper/LLM/TTS 自架推理伺服器）與團隊已有的自建管線高度重疊卻沒有額外優勢，且近期維護節奏放緩、社群案例遠少於 xiaozhi，整體槓桿低於 xiaozhi-esp32-server。不建議投入。

### 1.6 ESPHome Voice / Home Assistant Voice PE 生態

- **定位**：非一個可直接拿來跑的「專案」，而是 Home Assistant 官方的語音硬體與韌體生態——`esphome/home-assistant-voice-pe` 是官方語音喇叭裝置的韌體，`voice_assistant` 是 ESPHome 的元件，`microWakeWord` 是其官方推的輕量喚醒詞模型（基於 Google Inception 架構，可在 ESP32-S3 上跑）。
- **授權**：`esphome/home-assistant-voice-pe` 標示 `NOASSERTION`（GitHub 授權偵測未能明確判定，通常代表 LICENSE 檔案存在但格式非標準模板或含客製條款），使用前需另外人工確認條款細節。
- **活躍度**：704 stars、最後 push 2026-07-01，活躍；生態內各元件（ESPHome 本體、microWakeWord）也持續維護。
- **架構亮點可借鏡**：雙處理器音訊架構（ESP32-S3 負責 ESPHome 語音助理整合、XMOS 晶片做硬體加速語音前處理如回音消除/波束成形）、以及 microWakeWord 這種**專為低功耗 MCU 設計、模型小到能在 ESP32-S3 上跑的喚醒詞方案**，若團隊未來要在 Genio 520 上做喚醒詞（而非純 push-to-talk），這類輕量模型的設計思路（而非直接搬模型，因為目標晶片不同）值得參考。
- **對本專案的適配結論：淘汰（作為整包基底），僅列為架構設計參考**。整個生態系統的「大腦」預設是 Home Assistant，裝置端只是「衛星」，與說說學伴「單機、無中控、離線可運作、以 LLM 鷹架對話為核心」的產品定義完全不同方向，沒有整包沿用的空間。

## 2. 核心判斷題：xiaozhi-esp32-server 改造 vs 自建，28 天內誰完成度更高

### 2.1 結論先講：**維持自建為主線，不整包 fork 任何候選**

理由拆解如下：

| 面向 | xiaozhi-esp32-server 現成提供 | 說說學伴實際需要 | 差距是否為本專案核心差異化 |
|---|---|---|---|
| VAD / 音訊擷取狀態機 | 有（server 端內建） | push-to-talk 半雙工（比全雙工 barge-in 更簡單） | 否——團隊自建原型已用 asyncio 狀態機做同等級的事，且 push-to-talk 場景比 xiaozhi 預設的「隨時可打斷」場景更簡單，等於還要「拆掉」一部分 xiaozhi 的複雜度 |
| ASR/LLM/TTS 插拔介面 | 有（成熟 provider base class） | 需要接 NeuroPilot TFLite（NPU）與 llama.cpp（CPU）自訂 provider | 否——插件*介面設計*值得抄，但 NeuroPilot provider 本體無論如何都要自己從零寫，xiaozhi 生態圈沒有人做過 |
| WebSocket 協定/佇列管理 | 有（完整文件化協定） | 單機單裝置，client/server 甚至可能同機 loopback | 部分——協定設計可抄，但「多裝置佇列管理」這種 server 要服務多台 ESP32 的能力對單機玩偶是用不到的多餘複雜度 |
| 記憶系統 | 有（對話記憶，短期+mem0ai+PowerMem） | 學習狀態追蹤（錯題、難度、進度），非對話記憆 | **是**——資料模型完全不同，需要重新設計，只能抄「有個記憶模組」這個概念，schema 要重寫 |
| 教學鷹架引擎（多輪追問、難度分級、鷹架話術） | **無** | 核心產品邏輯 | **是，且是本專案最核心的差異化，必須從零建置** |
| 學習評分/評量 | **無** | 核心產品邏輯 | **是，必須從零建置** |
| 教師端資料出口/儀表板 | 有 manager-web/manager-api 但功能是「設備管理」非「教學報表」 | 教師儀表板（雲端助教層） | **是**——可能可以參考 Vue+後台的專案骨架省一點前端 boilerplate，但資料模型、報表邏輯要全部重寫 |
| 兒少內容安全層 | **無**（通用聊天玩具，無審核機制） | 核心合規需求 | **是，必須從零建置** |
| NPU/ARM Yocto 部署驗證 | **無**（官方部署鎖定一般 Ubuntu server，非 Yocto，也無 NeuroPilot 案例） | 核心技術風險 | **是**——無論選誰，Yocto 上的 Python 套件生態、NeuroPilot Delegate 整合都要自己趟出一條路，xiaozhi 在這塊完全沒有現成案例可循，團隊自建原型至少已經在朝這個方向驗證 |

**若要採用 xiaozhi-esp32-server 改造，需要新增/重構的清單**（對應題目要求）：
1. **教學 prompt / 鷹架引擎**：xiaozhi 的「角色設定」只是靜態 system prompt，需要新增一層鷹架狀態機（追問策略、難度升降、code-switching 提示詞模板），這與自建方案工作量完全一致，沒有省到力。
2. **評分/學習追蹤**：需要新建 SQLite schema（哪些字詞錯過、發音評分、完成度）與對應的 LLM 後處理邏輯，xiaozhi 記憶系統無法沿用，等同重新設計。
3. **教師端資料出口**：manager-web/manager-api 架子（Vue+Java SpringBoot）可以省一些前端元件與 CRUD boilerplate，但資料模型、報表頁面幾乎要重做；且多學一套 Java 後台在 28 天內對熟悉 Python 的團隊是額外學習成本。
4. **兒少內容安全層**：需要整個新增 pre/post-filter（關鍵字＋LLM 分類器），xiaozhi 完全沒有對應機制。
5. **NPU/Yocto 整合**：無論選誰都要自己寫 NeuroPilot provider、驗證 Yocto 上的套件安裝方式，這塊工作量兩案相同、不受選型影響。

**額外要付出但不一定回收的成本**：讀懂一個 3,652-commit、你不熟悉的外部程式碼庫（即使核心 Python 部分不大，仍需要時間建立 mental model、確認哪些是 ESP32-only 的假設、哪些程式碼路徑是為多裝置/雲端 API 場景設計而用不到），以及後續每次上游更新/修 bug 時要花心力判斷「這個改動會不會影響我魔改過的部分」。這對一個已經有能跑的自建原型、且核心差異化功能都要自己寫的專案來說，**投資報酬率偏低**。

### 2.2 建議的折衷做法（抄設計不抄程式碼）

1. 花 **0.5–1 天**精讀 `xinnan-tech/xiaozhi-esp32-server` 的 `main/xiaozhi-server` 核心與其 [WebSocket 協定文件](https://github.com/78/xiaozhi-esp32/blob/main/docs/websocket.md)，抄下協定訊息 schema 設計、ASR/LLM/TTS provider base class 的介面切法，套用到自建 FastAPI 專案的模組介面設計上（讓未來要換 ASR/TTS 引擎時更省力）。
2. 記憶系統的「短期 + 長期」分層概念可作為教學進度追蹤資料模型設計時的參考架構（但 schema 要重寫成教學語意）。
3. 不 fork、不整包部署，不引入 Java/Vue 管理後台依賴。

## 4. 28 天內整合路徑（自建為主線 + 選擇性抄設計）

- **Day 1–2**：花半天精讀 xiaozhi-esp32-server 的 WebSocket 協定文件與 provider 插件介面設計（僅供設計參考，不 clone/不部署），將可用的介面切法（ASR/LLM/TTS base class）套進自建 FastAPI 專案的模組邊界，方便後續替換元件（例如未來把 faster-whisper 換成 NeuroPilot TFLite ASR）。
- **Day 3–7**：延續自建管線在 Genio 520/Yocto aarch64 上的驗證與優化（NeuroPilot Neuron Delegate 整合、llama.cpp CPU 推論延遲調校），此階段工作量與是否採用 xiaozhi 無關，兩案皆須投入。
- **Day 8–14**：建置**教學鷹架引擎**（多輪追問策略、難度分級、中英夾雜提示詞模板）與**學習評分/追蹤**資料模型（SQLite schema，記錄錯題、發音評分、進度），可參考 xiaozhi 記憶系統「短期+長期」分層概念設計資料結構，但邏輯完全重寫。
- **Day 15–19**：**兒少內容安全層**（關鍵字 pre-filter + LLM 分類器 post-filter）、半雙工 push-to-talk 狀態機打磨、兜底話術。
- **Day 20–24**：雲端助教層（AWS Bedrock/Claude 對接、每學生持久記憶 agent）與**教師儀表板**（資料出口 API + 前端；若時間緊繃可考慮參考 xiaozhi manager-web 的 Vue 元件庫節省前端 boilerplate，但不建議整套後台架構搬過來，避免多學一套 Java SpringBoot）。
- **Day 25–28**：整合測試、離線斷網 demo 演練、延遲調校（TTS 首音 <300ms／單輪 <2s 目標驗證）、備援兜底話術覆蓋率檢查。
